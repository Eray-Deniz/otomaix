Bu rapor, /root/otomaix/apps/social dizinindeki FastAPI backend ve Next.js 14 frontend uygulamasının teknik analizidir. Amaç: şu ana kadar yapılan mimari çalışmaları özetlemek, çalışmayan / eksik / mantık hatası içeren bileşenleri tespit etmek ve öncelikli bir yapılacaklar listesi çıkarmaktır.

1. Genel Mimari Özeti

Platform; Türk KOBİ ve orta ölçekli şirketlere yönelik, AI destekli sosyal medya içerik üretim ve yayın otomasyonu sunan bir SaaS ürünüdür. Üç ana katman mevcuttur:

  • Frontend (app.otomaix.com) — Next.js 14 App Router, shadcn/ui, Tailwind, Zustand store, Supabase Auth, FullCalendar, recharts, PostHog, Sentry, Crisp.
  • Backend (api.otomaix.com) — FastAPI + asyncpg (PostgreSQL 5433, pgvector), Redis cache & rate-limit, Cloudflare R2 (assets.otomaix.com), fal.ai (FLUX.2 Pro + Kling), Anthropic Claude, Upload-Post.com (sosyal yayın), Paddle (ödeme), Apify (rakip analizi).
  • Otomasyon (n8n.otomaix.com) — Auto-posting scheduler, Telegram onay akışı, Türkiye takvim güncelleme, haftalık rakip raporu gibi workflow'lar.

Backend kod tabanı 17 router + 10 servis, ~4.900 satır Python. Frontend 12 dashboard sayfası, ~6.130 satır TSX.

2.2 Kritik Bulgular (Backend)

[B-1] Plan limit kontrolü hiçbir yerde çağrılmıyor.

billing.py:515 içindeki check_plan_limit() fonksiyonu tanımlanmış fakat posts.py, brands.py, trends.py ve avatar.py içindeki hiçbir endpoint'te bu fonksiyon çağrılmıyor. Sonuç: Starter planındaki bir kullanıcı aylık 50 içerik limitini aşabilir, 1 marka limitinin üzerinde marka açabilir, Pro'ya özel video üretimini Starter'da kullanabilir. Ödeme modelinin finansal etkisi var.

Yapılması gereken: /posts/generate, /posts/generate-faceless-video, /brands (POST), /avatar/generate-ugc endpoint'lerine await check_plan_limit(...) eklenmeli.

[B-2] Yetkilendirme (ownership) kontrolü çoğu endpoint'te yok.

Supabase JWT doğrulaması yapılıyor, ancak kullanıcının istek yaptığı kaynağın sahibi olup olmadığı kontrol edilmiyor. Örnekler:

  • GET /brands/{brand_id} — herhangi bir JWT ile herhangi bir markanın brand_kit'ine, logo URL'sine, sektör bilgisine erişilebilir.
  • PATCH /brands/{brand_id}, DELETE /brands/{brand_id}, PATCH /brands/{brand_id}/kit — başkasının markasını silebilir / değiştirebilir.
  • POST /posts/{id}/publish, POST /posts/{id}/regenerate, GET /posts/{id} — başkasının içerikleri üzerinde işlem yapılabilir. (Sadece /posts/{id}/request-approval kullanıcı–workspace–brand join'i yapıyor.)
  • /calendar/posts, /trends, /competitors, /autoposting/config — brand_id parametresi sahiplik üzerinden doğrulanmıyor.
  • GET /posts?brand_id — başka markaların içerik listesi okunabilir.

Yapılması gereken: Ortak bir assert_brand_owned(user_id, brand_id, db) yardımcısı yazılıp kaynak bazlı endpoint'lerde dependency olarak kullanılmalı.

[B-3] autoposting.py şeması Telegram alanlarını yazmıyor — CLAUDE.md ile tutarsız.

CLAUDE.md: "Her müşteri kendi bot token'ını Otomatik Yayın wizard'ına girer → social.autoposting_configs.telegram_bot_token / telegram_chat_id kolonlarında saklanır."

Gerçek kod (autoposting.py:17–72): AutopostingConfigRequest şemasında bu alanlar yok, INSERT cümlesi de bu kolonlara değer yazmıyor. Frontend otomatik-yayin/page.tsx de artık bu alanları göndermiyor ve kullanıcıyı /ayarlar'a (workspace Telegram ayarı) yönlendiriyor.

Durum: Telegram artık workspace seviyesinde saklanıyor (social.workspaces.telegram_bot_token / telegram_chat_id). internal.py:get_due_configs ve posts.py:request_approval bu kolonları workspaces tablosundan okuyor (doğru). CLAUDE.md güncellenmeli — ayrıca autoposting_configs'daki eski kolonlar kullanılmıyor, migration 004/005 sonrası temizlik gerekiyor.

[B-4] competitors.py add_competitor — ölü kod + potansiyel JSONB hatası.

competitors.py:62-71 aralığında, run_full_analysis() sonrası önce yanlış bir UPDATE yapılıyor:

UPDATE ... SET analysis_data = $2 ... VALUES (str(analysis_data).replace("'", '"'))

Python str(dict) çıktısı valid JSON değildir (True/False/None olarak kalır), üstelik ::jsonb cast'i yok. asyncpg'nin bu sütunu (JSONB) ham string olarak kabul etmesi tip hatasına yol açar. Hemen ardından (73-82) doğru json.dumps() + ::jsonb UPDATE yapılıyor — yani ilk UPDATE ölü koddur ve potansiyel olarak exception fırlatıp ikinci UPDATE'i engeller. İlk UPDATE bloğu tamamen silinmeli.

[B-5] competitors.py ve trends.py senkron bloklama.

POST /competitors istek boyunca run_full_analysis() çağrısı yapıyor: website HTTP fetch + Claude Haiku + (varsa) Apify Instagram — tamamı tek istek içinde senkron. Bu istek kolayca 30+ saniye bloklayabilir, uvicorn worker'ı tüketir. Ayrıca Cloudflare/Coolify default timeout'u (genelde 60sn) aşılabilir.

Yapılması gereken: FastAPI BackgroundTasks veya arka plan kuyruğu (Celery/RQ) ile asenkron işleme; 202 Accepted dönülüp frontend polling yapmalı.

[B-6] trends.py:refresh_trends UPSERT değil INSERT yapıyor.

trends.py:84 her refresh'te aynı sektör için yeni bir satır ekliyor. trend_cache tablosu zamanla şişecek. ON CONFLICT (sector) DO UPDATE ya da periyodik temizlik gerekli.

[B-7] brands.py:update_brand_kit — brand_kit JSONB string/dict ayrımı yok.

posts.py yardımcı _parse_brand_kit() fonksiyonu ile asyncpg'nin bazen JSONB'yi string döndürdüğü durumu handle ediyor. brands.py:132 ise doğrudan dict(row['brand_kit']) yapıyor — değer string geldiğinde TypeError atacak. Aynı parse helper brands.py'de de kullanılmalı.

[B-8] posts.py:publish_post — durum kontrolü yok.

POST /posts/{id}/publish çağrısı post'un ready durumunda olup olmadığını kontrol etmiyor. generating, failed, published bir post tekrar publish edilmeye çalışılır ve servis tarafında output_url kontrolünde ValueError fırlar → 500 döner. Frontend butonu disabled yapıyor ama backend idempotency / durum validasyonu yapmalı.

[B-9] upload_post.py — plan features ignore ediliyor.

Servis tüm platformlara sırayla POST /publish çağrıları yapıyor; all_ok True değilse post 'failed' oluyor ve başarılı yayınlar da sıfırlanmış gibi görünüyor. Kısmi başarı (3 platformdan 2'si OK) kaybediliyor. Önerilen: social.post_publications tablosu (post_id, platform, status, external_id, error) oluşturup platform bazlı durum tutmak.

[B-10] /internal/posts/scheduled-due uç noktası var — ancak eşlenmiş n8n workflow belgelenmemiş.

internal.py:213'te Scheduled Post Publisher için endpoint hazır (scheduled_at geçmiş 'scheduled' post'ları döndürür). Ancak CLAUDE.md'deki n8n workflow listesinde bu endpoint'i 5 dakikada bir çağıran bir workflow yok. Sonuç: Takvime zamanlanan içerikler otomatik yayınlanmıyor olabilir. Doğrulanmalı + eksikse n8n workflow oluşturulmalı.

6. Sonuç ve Tavsiyeler

Genel tablo: Dört fazın tamamı tamamlanmış, mimari sağlam ve modüler. Backend router ayrımı temiz, fal.ai + R2 + Upload-Post zinciri webhook tabanlı ve doğru kurulmuş, Paddle/CRM webhook'ları hazır, Redis cache/rate-limit uygulanmış, Sentry + PostHog gözlenebilirlik aktif. Frontend App Router + Zustand + shadcn temeli oturmuş; Marka Ayarları, Kütüphane, Takvim, Otomatik Yayın, Trendler, Rakip Analizi, Faturalandırma gibi sayfaların hepsi tam işlevsel.

Ancak ürünü bugün canlıya açtığınız an dört kategori problem ortaya çıkar:

  • Finansal risk: Plan limitleri uygulanmıyor — kullanıcılar Pro özellikleri Starter'da sınırsız kullanabilir (B-1).
  • Güvenlik: Ownership kontrolü yok; workspace/marka kaynakları JWT sahibi herkes için okunabilir (B-2).
  • Kullanıcı akışı: İçerik Oluştur wizard'ından üretilen içerik için tek çıkış "kütüphaneye git → tekrar tıkla" — Step 3'teki yayınla/zamanla butonları bilerek toast gösteriyor (F-3, F-2, F-1).
  • Otomasyon katmanı ölü: Auto Posting Scheduler inaktif ve yanlış URL ile yapılandırılmış (N8N-2/3), Telegram onay butonları runtime'da patlıyor (N8N-1), scheduled post publisher hiç yok (N8N-7), CRM-4/5 Postgres'e ulaşamıyor (N8N-4). Otomatik yayın özelliğini müşteri "açsa bile" hiçbir şey çalışmıyor.

İlk sprint (1 hafta) önerisi: P0 maddeleri. Bu beş maddeden sonra ürün ticari anlamda sağlam, ikinci sprint P1 ile UX'i cilalarsınız.

7.2 Tespit Edilen Kritik Sorunlar

[N8N-1] KRİTİK — Telegram Onayla/Reddet workflow'ları runtime'da patlıyor

Canlı execution #51 (2026-04-13 20:30) aQ8neGzs3PQp8DMl için hata üretti: "Unused Respond to Webhook node found in the workflow". Aynı yapılandırma 9kp6bCFl0ys6TbVu (Reddet) workflow'unda da mevcut. Webhook node'unun responseMode: responseNode moduna geçirilmiş olmasına rağmen ilk HTTP Request node'u continueOnFail: true + hata yolu olmadığı için n8n validator Respond to Webhook'u "unused" kabul ediyor. Sonuç: müşteri Telegram'da Onayla'ya basınca işlem çalışmıyor, publish-now asla tetiklenmiyor.

Not: execution #53 (2026-04-13 20:33) başarıyla tamamlandı — hata aralıklı görünüyor. Yine de kök neden (HTTP Request hata yolu eksikliği) kalıcı düzeltilmeli.
